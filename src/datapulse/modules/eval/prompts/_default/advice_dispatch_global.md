# 维度：BU 分发问题诊断（全局）

下面是本次评测中，问答系统在**分发**上的表现数据。分发有两类错误：
- **漏收**：本应由本 BU 承接的问题，却被拒识 / 分给了别处（该分未分）。
- **误收**：本不该本 BU 承接的问题，却被收了进来（该拒未拒）。

`dispatch_reason` 是评测模型对「该不该本 BU 承接」给出的判断依据。

## Payload 字段说明

下面 `{payload}` 里会给到这些数据（供你分析用，不要原样复述）：

| 字段 | 含义 |
| --- | --- |
| 分发准确率 | 本 BU 分发判定正确的比例 |
| 漏收（该分未分）总数 | 本应本 BU 承接却被拒/分走的问题数 |
| 误收（该拒未拒）总数 | 不该本 BU 承接却被收进来的问题数 |
| 高频被漏收问题 | 被漏收次数最多的问题清单（问题, 次数） |
| 高频被误收问题 | 被误收次数最多的问题清单（问题, 次数） |
| 漏收样例 / 误收样例 | 代表性样例，每条含：问题、AI 答案、分发判断依据 |

## 数据来源（这些样本是怎么筛出来的）

以上数据来自本次评测逐条落库的明细表 `t_eval_task_row`，按分发场景 `dispatch_scene` 筛选：
- **漏收（该分未分）** = 评测模型认为该本 BU 承接（`should_dispatch_to_bu=true`），但日志实际没分给本 BU（`dispatched_to_bu=false`）。
- **误收（该拒未拒）** = 评测模型认为不该本 BU 承接（`should_dispatch_to_bu=false`），但日志实际分给了本 BU（`dispatched_to_bu=true`）。
- 分发判断正确的行（`dispatch_scene=正常`）不在此列，因为它们没有分发问题。
