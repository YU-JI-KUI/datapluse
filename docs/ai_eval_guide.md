# AI 评测模块 · 使用与原理说明

> 面向业务/运营/评测同学。读完你会知道：**每个数字怎么算出来的、你改哪里会影响哪个数字、
> 改完要不要重跑**。

---

## 0. 一句话理解

上传一份对话日志 Excel → 系统对每条客户问题跑一次「AI 裁判（Judge）」→ 汇总成一份带
**分发准确率 / 问题解决率 / 业务分类分布** 等指标的评测报告；你可以配置规则少调模型、
人工复核纠偏、调提示词后重跑验证。

```
上传 Excel
   │
   ├─(过滤) 活动标问        → 整条跳过，不评、不计指标
   │
   ├─(短路) 规则命中        → 用写死结果，免调 LLM，但【计入指标】
   │
   └─(评测) 其余样本 ──► AI Judge ──► 逐条结果落盘 ──► 汇总指标
                                              │
                              ┌───────────────┼────────────────┐
                          人工复核         调优业务知识        导出/查看
                       (覆盖判定,          (改→试跑→保存        (报告/明细/
                        实时改指标)          →勾选重跑)          三种 Excel)
```

## 1. 术语表（先看懂这些词）

| 术语 | 是什么 | 说人话 |
|---|---|---|
| **BU（业务单元）** | securities=证券 / life=寿险 | 你在评测哪条业务线。分类、规则、提示词都按 BU 各存一套。 |
| **Judge（AI 裁判）** | 大模型对单条问答做的结构化判定 | 它输出 11 个字段：该不该本 BU 接、业务分类、是否解决、要不要人工复核… |
| **样本 / 明细行** | 日志里的一条「客户问题 + AI 答案」 | 评测的最小单位。一通多轮对话有多条样本。 |
| **分发（Dispatch）** | 这条问题该不该由本 BU 承接 | AI 判「该接/该拒」，和日志里「实际分给了谁」比对。 |
| **进漏斗 / dispatched_to_bu** | 日志「分发BU」列显示**实际分给了本 BU** | 只有进漏斗的样本才算「问题解决率」的分母。 |
| **金标（Gold）** | 人工事先在 Excel 里标的「是/否」列 | 有金标 → 校准模式（算 κ/F1 验证 AI 准不准）；没有 → 生产模式。 |
| **活动标问** | 前端写死按钮触发的写死回复（如「转人工」） | 不经 AI，评测时整条**跳过**。 |
| **规则短路** | 结果确定的问答，配好写死判定 | 命中就**免调 LLM**直接给结果，但**仍计入指标**。 |
| **业务知识** | 业务方自己写的补充判定规则 | 一段自然语言，拼进 Judge 提示词，让判定更贴业务。 |
| **人工复核** | 人工覆盖 AI 的某条判定 | 你说了算，指标按「人工优先」实时重算。 |
| **闭集分类** | 业务分类只能是你配的那几个 + 一个兜底桶 | 模型自创的分类统一归「非XX业务」，报告不会冒出几十个分类。 |

## 3. 完整流程 + 每个环节对指标的影响

### 阶段 A · 数据加载（不调 LLM）

上传的 Excel 必须含这些列（列名按「包含匹配」，缺必需列直接报错）：

- **必需**：`应用会话ID`、`客户咨询轮次`、`客户问题`、`答案`、`分发BU`
- **金标（可选）**：`分发是否正确`、`答案是否解决客户问题`（填「是/否」，决定是否进校准模式）

> ⚠️ **`分发BU` 是必需的**：分发准确率、问题解决率都以「日志实际分给谁」为事实侧。缺它
> 这些核心指标会全部失真，所以缺列直接报错、不放行。

**这一步影响的数**：
- `日志总条数` = Excel 行数。
- **活动标问过滤**：客户问题**精确等于**活动标问清单里的某条 → 整条跳过（不评、不计任何指标），
  但仍作后续轮的上下文保留。→ `评测样本数 = 日志总条数 − 活动标问数`。

### 阶段 B · 逐条评测（规则短路 + AI Judge）

对每条参评样本：

1. **先过规则短路**：客户问题**精确等于**规则的 `question` **且** 答案**精确等于** `expected_answer`
   → 命中，直接用规则里写死的 judge 结果，**不调 LLM**。命中结果**照常落盘、计入所有指标**
   （标记来源=rule）。答案对不上 → 不命中，走 AI。
2. **未命中 → AI Judge**：用本 BU 的提示词（system + 各任务规则 + 业务知识 + 分类清单）让大模型
   判定，产出 11 字段结果。
3. **业务分类闭集**：模型返回的业务分类，在清单内保留；清单外自创的、或占位值（非本BU/拒识/空）
   → 统一归 **「非XX业务」** 兜底桶。**报告业务分类恒为「你配的 N 个 + 非XX业务」**。
4. 逐条结果落盘到 `t_eval_task_row`。

**这一步影响的数**：分发判定、业务分类、是否解决、需复核——**几乎所有指标的原始数据都在这一步产生**。

### 阶段 C · 汇总指标

跑完后汇总成报告。各指标口径见下节。

## 4. 指标口径（每个数字怎么算的）★重点

### 📊 评测来源分布（整份日志拆三类）
```
日志总条数 = 活动标问数 + 规则命中数 + AI 评测数
             (跳过不评)   (免LLM,计指标)  (真正调LLM)
```

### 📊 BU 分发准确率
```
分发准确率 = 判对数 (correct) ÷ 参与评分数 (scored)
```
- **判对**：AI 判「该不该本 BU 接」与日志事实「实际是否分给本 BU」**一致**（该接且接了 / 该拒且拒了）。
- **参与评分数**：AI 给出了有效分发判定的样本数（**模型出错/缺字段的不计入分母**）。
- **两类错误**（细分，占比分母同为 scored）：
  - **漏收**：AI 判该接，实际没分到本 BU。
  - **误收**：AI 判该拒，实际却分到了本 BU。
- 用 AI 判断 vs 日志客观事实比对，**不依赖人工金标**，故生产模式也能算。

### 📊 问题解决率
```
问题解决率 = 已解决数 ÷ 实际分到本 BU 的样本数（进漏斗数）
```
- **分子**：AI 判 `answer_resolved = yes` 的样本（partial/no 都不算已解决）。
- **分母**：**只算进漏斗（dispatched_to_bu=True）的样本**——拒识/没分到本 BU 的不计入分母。

### 📊 需人工复核数
- AI 自判 `needs_human_review = true` 的条数（触发：疑似合规风险 / 信息严重不足 / 明显自相矛盾）。
- **被人工复核后 −1**（无论复核有没有改判定，都视为「已人工确认」，从需复核队列移除）。

### 📊 业务分类分布
- 按 AI 打的业务分类计数；闭集：只出现你配的分类 + 「非XX业务」兜底桶。

### 📊 会话轮次分布（按会话，不是按样本）
- 每通对话的**总轮次** = 该会话内最大轮次（一通 5 轮对话 = 1 个「5 轮会话」）。
- 动态分桶：1轮/2轮/3轮 各单独档 + 高轮次按中位数动态切 2 档，**区间由数据定、不写死**，≤5 档。

### 📊 校准指标 κ / F1（仅校准模式）
- **何时算**：Excel 的 `分发是否正确` 或 `答案是否解决客户问题` 列里有「是/否」标注 → calibration 模式。
- **κ（Cohen's Kappa）**：衡量「AI 判定 vs 人工金标」的一致性，扣除了「碰巧一致」。κ=1 完全一致、
  κ=0 等于瞎猜。比准确率更严格，样本不均衡时看它更可信。

## 5. 三种「纠偏/调优」操作，指标怎么变 ★重点

这是最容易混的地方。**核心规律**：

> **改「判定规则」类的东西（提示词/业务知识/分类/规则/活动标问）→ 只影响下次评测，
> 已有报告不会自动变，要重跑才反映。**
> **只有「人工复核」是例外——它是叠加层，实时改指标，不用重跑。**

| 操作 | 在哪 | 调 LLM? | 落库? | 指标变化 | 何时生效 |
|---|---|---|---|---|---|
| **单条试跑** | 详情抽屉 | 是(1条) | 否 | 不变 | 即时预览，纯验证提示词改得对不对 |
| **人工复核** | 详情抽屉 | 否 | 是 | **增量重算**(实时) | **立即**，重进结果页就是新指标 |
| **勾选重跑** | 明细表任意视图勾选 | 是(选中的) | 是(覆盖) | **全量重算** | 后台异步（任务转「重跑中」显进度），完成后重进结果页看新指标；跳过已复核，上限 2000 条 |
| **全量重测** | 历史/结果页 | 是(全部) | 是(清空重跑) | 全部重新评 | 数小时 |

### ① 人工复核（改一条的最终结论，实时改指标）
- 详情抽屉的「人工复核」区：改「分发是否正确 / 是否解决 / 业务分类」+ 写评论，提交。
- 指标按**最终值**（人工优先、AI 兜底）**增量重算**：
  - 分发：复核值与 AI 不同 → correct ±1（scored 不变；漏收/误收细分保持 AI 原值）。
  - 解决：仅对进漏斗的行生效，复核值与 AI 不同 → 已解决数 ±1。
  - 需复核：被复核的行从需复核数扣 1。
- **不用重跑**，重进结果页指标就变了。撤销复核可还原。

### ② 调优业务知识（改判定规则，让 AI 判得更准）
业务人员在**详情抽屉**里的闭环：
1. 看到 AI 判错 → 在「调优业务知识」区**当场编辑**本 BU 的业务知识（补规则，如「含'已为您转接
   人工'的答案视为已解决」）。
2. **试跑**：用编辑框里的**临时内容**（还没保存）重评这一条，看新旧对比。不落库、不影响指标。
3. 不满意继续改；满意 → **保存业务知识**（写入库，整个 BU 共享）。
4. **保存 ≠ 指标变了**——业务知识改的是判定规则，要让它作用到已有报告，必须：
   - **勾选重跑**（在明细表任意视图勾选要重评的行 → 点「用最新提示词重跑选中(N)」，后台
     异步跑、跳过已复核、完成后全量重算指标；上限 2000 条）；或
   - **全量重测**（整个任务重跑）。

### ③ 提示词管理（更技术的判定规则）
`/eval/prompts` 各槽位影响不同维度，改后**下次评测生效**，已有报告需重跑：

| 槽位 | 影响 |
|---|---|
| `judge_system` | AI 的人设/总体判定立场 |
| `task_dispatch` | 分发判定 → 影响分发准确率 |
| `task_business_type` | 业务分类打标 → 影响分类分布 |
| `task_resolved` | 是否解决 → 影响解决率 |
| `task_review` | 需复核判定 → 影响需复核数 |
| `business_knowledge` | 业务方补充规则（业务人员改这里，别碰上面几个技术槽位） |

> 提示词有**快照隔离**：评测一开始就快照当前提示词，整个任务用同一份——跑到一半有人改
> 提示词，不会污染进行中的任务，只影响下次。

## 6. 典型工作流

**A. 首次评测**
1. `/eval` 选 BU、上传日志 Excel（或用样例）→ 起评测。
2. `/eval/history` 看进度 → 完成后进结果页看指标。

**B. 发现指标异常 → 定位 → 调优**
1. 结果页看「评测来源分布 / 分类分布 / 轮次分布」找异常。
2. 明细表筛「需复核 / 某分类」→ 点开详情抽屉看 AI 判错在哪。
3. 判错分两种：
   - **个别条判错** → 直接**人工复核**纠正（实时改指标）。
   - **一类系统性判错** → 详情抽屉**编辑业务知识 → 试跑验证 → 保存** → 明细表**勾选相关行重跑**（或全量重测）。

**C. 省 LLM 调用**
- 结果确定的高频问答 → `/eval/rules` 配规则短路（问题+答案匹配即免 LLM）。
- 写死按钮问题（转人工等）→ `/eval/activity` 配活动标问（整条跳过）。

**D. 导出**（结果页，大数据量有加载态、防狂点）
- 评估报告（多 sheet 概览）/ 逐条明细 / 不一致 case。文件名带上传文件名前缀。

## 7. 稳定性保障（了解即可）

- **多 POD 抢占式调度**：任务状态在 DB，`FOR UPDATE SKIP LOCKED` 抢任务 + 全局串行锁（同时只跑
  一个评测，不压垮内网 LLM 网关）+ 心跳续约 + 崩溃自动接管续跑。
- **断点续跑**：任务中断（失败/限流/进程重启）后，只补未完成的行，已落盘不重做。
- **限流退避**：撞大模型限流自动暂停、退避后续跑；一批里已成功的先落盘。
- **坏 JSON 兜底**：模型偶发吐非法 JSON → 升温重试一次 + 解析容错。

---

## 8. 一句话速查

- **改分类/规则/活动标问/提示词/业务知识 → 只影响下次评测；旧报告要重跑（子集/全量）才变。**
- **人工复核 → 实时改指标，不用重跑。**
- **单条试跑 → 只预览，不落库、不动指标。**
- **分发准确率分母**排除模型出错的；**解决率分母**只算进漏斗（分到本BU）的。
- **业务分类永远是闭集**：你配的 N 个 + 「非XX业务」。
