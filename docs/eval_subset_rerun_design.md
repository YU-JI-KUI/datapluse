# 评测子集重跑设计（业务知识调优后重算指标）

## 背景与问题

业务人员在详情页微调「业务知识」提示词后，只更新了 prompt（库里），**已跑完的报告
指标不会变**——因为分发准确率/解决率/需复核数等是评测时用**旧 prompt** 算好、落盘在
`result_json` 的静态快照。

要让业务知识改动体现在指标上，**必须用新 prompt 重新 judge 样本**（这一步省不掉 LLM
调用，区别只在重跑多少条）。全量重跑（现有 rerun）5万条要几小时且大量浪费——业务知识
通常只影响少数样本。故做「子集重跑」：只对用户选定的一批样本用最新 prompt 重评。

## 核心限制（必须认清）

业务知识是自然语言规则，**系统无法自动判断"哪些样本会被影响"**（要 LLM 判定才知道）。
所以"受影响子集"由**用户按可见维度手动选定**：需复核 / 某业务分类 / 不一致 / 关键字。
用户重跑哪批，就重算哪批；这批之外的判定不变。

## 已确认的决策

| 点 | 决策 |
|---|---|
| 重跑范围 | 用户按筛选维度手动选一批（需复核/某分类/不一致/关键字） |
| 结果处理 | 重跑结果**覆盖** `t_eval_task_row` 对应行 |
| summary 指标 | 重跑后**全量重算**（扫全部 row 重新聚合，不调 LLM，绝对准确） |
| 复核冲突 | **跳过已人工复核的行**（人工结论优先，不被自动覆盖） |

## 数据流

```
用户在明细页某筛选视图（需复核/某分类/…）点「用最新提示词重跑这批」
  → 后端 submit 一个「子集重跑任务」到 worker（复用现有后台 worker）
      → 取这批 row 的 row_index（按筛选条件查，排除已复核的）
      → 分批：从 row_json 重建 sample → 用当前 bu（最新业务知识）judge_batch
      → 覆盖 t_eval_task_row 对应行（row_json 更新，标 _rerun_at）
  → 全部重跑完 → 全量重算 summary（扫 t_eval_task_row 聚合）→ 覆盖 result_json
  → 前端刷新，指标反映新 prompt 下这批的结果
```

## 实现拆解

### 1. 子集选择（哪些 row 重跑）
复用现有筛选：
- **needs_review**：`row_json->judge->>needs_human_review = true`
- **某业务分类**：`row_json->>j_intent = X`
- **不一致**：`row_json->>is_disagreement = true`（有金标时）
- **关键字**：`row_json->>question ilike %q%`

统一：`select row_index where <筛选> and row_index NOT IN (t_eval_review 该任务)`
（排除已复核）。返回待重跑的 row_index 列表。

### 2. 重跑执行（复用现有基础设施）
- 复用 `assemble_row_sample_from_row`（已有，单条试跑在用）从 row_json 重建 sample。
- 复用 `judge_batch`（已有并发+限流+落盘那套）对这批 sample 重评。
- 用 `get_bu(bu_code)`（已注入最新业务知识快照）。
- 覆盖：`save_rows` 对这些 row_index upsert（已有，主键 (task_id,row_index) 覆盖）。
- 走**后台 worker**（复用 DB 抢占式那套）：子集重跑也是长任务，不能卡 web 请求。
  → 需要一个新的「子集重跑」任务类型，或给 eval_worker 增加一种 job。

### 3. summary 全量重算
- 重跑覆盖完 row 后，扫全部 `t_eval_task_row` 用 `_StreamAggregator` 重新聚合一遍，
  产出新 summary/insights/intent_distribution/bu_dispatch，覆盖 result_json。
- 这一步不调 LLM，纯读表 + 内存聚合（5万行分批读，和现有续跑读回一样）。
- 复核调整（apply_reviews_to_result）在 get_result 时照旧叠加，口径一致。

### 4. 复核冲突
- 子集选择时 `NOT IN t_eval_review` 已排除已复核行，从源头跳过。
- 保证：人工复核过的判定不被自动重跑冲掉。

### 5. 前端
- 明细页某筛选视图加「用最新提示词重跑这批（N 条）」按钮 + 二次确认（告知会调 LLM、
  覆盖 AI 判定、跳过已复核）。
- 重跑是异步任务：提交后轮询进度（复用任务进度机制），完成后刷新指标。

## 与现有能力的边界（避免混淆）

| 能力 | 触发 | 是否调 LLM | 是否落库 | 是否改指标 |
|---|---|---|---|---|
| 单条试跑（已有） | 详情页单条 | 是（1条） | 否 | 否（纯预览） |
| 人工复核（已有） | 详情页改结论 | 否 | 是（t_eval_review） | 是（增量叠加） |
| **子集重跑（本设计）** | 筛选视图批量 | 是（子集） | 是（覆盖 row） | 是（全量重算） |
| 全量重跑 rerun（已有） | 整个任务 | 是（全部） | 是（清空重跑） | 是（重新评测） |

## 风险与注意

1. **成本**：子集仍可能很大（如"某高频分类"有上万条）→ 前端确认框显示条数，让用户
   知道要调多少次 LLM。
2. **进度与中断**：子集重跑要能像正常评测一样断点续跑（复用 done_row_indices 机制？
   —— 但覆盖场景下 done 判断不同，需单独设计重跑任务的进度跟踪）。
3. **summary 重算时机**：必须在子集全部重跑完后再算，中途算会得到半新半旧的混合指标。
4. **并发**：子集重跑与全局 advisory lock 的关系——是否占用同一把"全局串行"锁。

## 分阶段建议

- **阶段一**：单一维度（先只支持"需复核"子集）重跑 + 全量重算 + 前端入口。跑通闭环。
- **阶段二**：扩展到任意筛选维度（分类/不一致/关键字）。
- **阶段三**：进度跟踪、断点续跑、大子集保护。
